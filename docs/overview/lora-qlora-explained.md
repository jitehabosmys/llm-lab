# LoRA 与 QLoRA 说明

这份文档专门解释当前项目里频繁出现的几个概念：

- `LoRA`
- `QLoRA`
- 量化与反量化
- `NF4`
- `bf16`

目标不是写成论文，而是帮助在当前项目语境下快速建立正确直觉。

相关文档：

- [config-rationale.md](/hy-tmp/llm-lab/docs/overview/config-rationale.md)
- [model-results-cheatsheet.md](/hy-tmp/llm-lab/docs/overview/model-results-cheatsheet.md)

---

## 1. LoRA 是什么

`LoRA` 的核心思想是：

**不去更新整套大模型参数，而是在某些线性层旁边额外挂一小组低秩参数，只训练这一小组增量。**

如果把原始线性层权重写成：

- `W`

那么 LoRA 不是直接把 `W` 改掉，而是学习一个增量：

- `ΔW = B @ A`

其中：

- `A` 的形状通常是 `rank × in_dim`
- `B` 的形状通常是 `out_dim × rank`
- `rank` 通常远小于 `in_dim` / `out_dim`

最终模型实际使用的是：

- `W + α * ΔW`

这里：

- `rank` 控制可学习增量的容量
- `alpha` 控制 LoRA 更新的缩放强度

所以 LoRA 的本质是：

- 基座模型大部分参数冻结
- 只训练一个低秩补丁

---

## 2. 为什么 LoRA 有用

LoRA 的价值主要有三点：

### 1. 显存更省

因为不需要给整套模型参数都维护梯度和优化器状态。

### 2. 训练更快

真正参与优化的参数量很小。

### 3. 产物更轻

保存出来的是 adapter，不是整个完整模型。

所以你会看到：

- 原始模型十几 GB
- 训练输出只有几百 MB 到几 GB

这是正常现象，因为保存的是 LoRA 增量，而不是整套 base model。

---

## 3. QLoRA 是什么

`QLoRA` 可以理解成：

**在 LoRA 的基础上，把基座模型先按低 bit 数量化加载，再在其上训练 LoRA adapter。**

也就是说：

- LoRA：基座模型通常还是 16-bit / bf16 权重
- QLoRA：基座模型先变成 4-bit（常见）再挂 LoRA

所以 QLoRA 的关键不是：

- “LoRA 参数也变成 4-bit 训练”

而是：

- **被冻结的基座模型以量化形式驻留显存**
- **LoRA adapter 仍然以更高精度参与训练**

这就是为什么：

- `8B / 14B` 在 4090 上还能训
- 而如果不用 QLoRA，单卡很可能就吃不消

---

## 4. 训练时到底哪些参数在更新

### LoRA

训练时：

- 基座权重 `W` 冻结
- `A` / `B` 这些 LoRA 参数更新

### QLoRA

训练时：

- 基座权重以量化形式加载，并冻结
- LoRA 参数更新

所以无论 LoRA 还是 QLoRA，真正训练的都是：

- adapter 参数

差别主要在于：

- 基座模型是普通精度加载
- 还是量化加载

---

## 5. 量化和反量化大致是怎么发生的

这里说的是直觉层，不追求每个底层 kernel 的严格实现细节。

### 第一步：量化存储

在 QLoRA 里，基座模型权重不再以 `fp16 / bf16` 直接保存到显存，而是先被压到更低 bit 宽度。

比如：

- 4-bit

这样模型本体占的显存会显著下降。

### 第二步：前向计算前的“临时还原”

虽然权重是低 bit 存的，但矩阵乘法不能真的用“抽象 4-bit 整数概念”直接做完整高质量计算。

所以实际前向时通常会发生：

- 从量化表示中读取权重
- 按某种规则还原到更高精度表示
- 再与输入激活做乘法

这个过程常被口语化说成：

- **反量化**

但更准确的理解是：

- 权重长期存储在压缩格式里
- 计算时临时转成合适的 compute dtype

### 第三步：LoRA 增量参与计算

在基座矩阵乘法之外，再加上 LoRA 低秩更新项：

- `x @ W_quantized`
- 再加上 `x @ ΔW_lora`

最后合在一起得到输出。

所以 QLoRA 的关键直觉是：

- **存储时尽量压缩基座**
- **计算时恢复到合适精度**
- **更新时只更新 LoRA**

---

## 6. 什么是 `NF4`

`NF4` 是 `NormalFloat4` 的缩写。

它是：

- 一种 4-bit 量化表示方式

为什么它常见于 QLoRA：

- 它比一些更朴素的 4-bit 表示更适合神经网络权重分布
- 对大模型做 4-bit 量化时，通常比简单 `fp4` 更稳

在当前项目里，你看到的：

```yaml
quantization_bit: 4
quantization_type: nf4
```

可以直觉理解为：

- 基座模型用 4-bit 压缩
- 具体采用 `NF4` 这种较常用、较稳的 4-bit 表示

---

## 7. 什么是 `double_quantization`

你配置里常见：

```yaml
double_quantization: true
```

它的含义可以直观理解成：

- 不只压缩权重本身
- 连量化过程中要用到的一些量化常数也进一步压缩

目的还是：

- 继续省显存

在实践里，它通常是一个很自然的 QLoRA 配置项，因为：

- 省空间
- 一般代价不大

---

## 8. 什么是 `bf16`

`bf16` 是 `bfloat16`。

它和 `fp16` 一样，都是 16-bit 精度格式。  
但它们不是同一种 16-bit。

`bf16` 的常见特点是：

- 指数范围更像 `fp32`
- 数值稳定性通常比 `fp16` 更舒服

所以在训练里你常看到：

```yaml
bf16: true
```

这表示：

- 前向 / 反向计算尽量使用 `bfloat16`

在当前项目里，`bf16` 常被当作一个比较稳妥的默认值，因为：

- 显存比 `fp32` 省
- 又比 `fp16` 更不容易因为数值范围问题出很多怪事

---

## 9. `NF4` 和 `bf16` 是什么关系

这两个经常一起出现，但它们控制的不是同一层东西。

### `NF4`

控制的是：

- **基座模型权重如何被压缩存储**

### `bf16`

控制的是：

- **训练 / 推理时主要采用什么计算精度**

所以你可以把它们理解成：

- `NF4`：怎么省空间存
- `bf16`：怎么比较稳地算

在 QLoRA 里，典型组合就是：

- 基座模型：4-bit `NF4`
- 计算：`bf16`
- 可训练的 LoRA 参数：高精度参与优化

---

## 10. 为什么 4B 用 LoRA，8B/14B 用 QLoRA

这就是资源约束下的工程选择。

### 4B

单卡 4090 上：

- 普通 LoRA 已经能比较从容地跑

所以：

- 没必要引入额外量化复杂度

### 8B / 14B

如果还按普通 LoRA 去加载完整高精度基座：

- 显存压力会明显变大
- 训练稳定性和承载空间都会更差

所以到这个规模，QLoRA 就变成更现实的方案。

---

## 11. LoRA / QLoRA 的局限

它们不是没有代价的。

### 1. 可表达能力受 rank 限制

如果：

- `rank` 太小

模型能学到的增量空间就有限。

### 2. QLoRA 会增加推理与加载复杂度

训练能跑，不代表推理部署链路也同样轻松。

尤其到了大模型上，你已经看到：

- 14B 训练可以
- 但 HF / PEFT / vLLM 推理链都开始碰边界

### 3. 不是所有收益都来自 LoRA 本身

当前项目里，收益往往还混合了：

- prompt 协议变化
- strict 主线收束
- 模型规模提升
- 数据质量提升
- DPO 偏好数据影响

所以不能把所有变化都粗暴归因给 LoRA / QLoRA。

---

## 12. 在当前项目里，一句话怎么理解

### LoRA

**给一个冻结的大模型挂上一个小补丁，只训练这个补丁。**

### QLoRA

**先把冻结的大模型压缩成 4-bit 存在显存里，再训练这个小补丁。**

### `NF4`

**一种更适合神经网络权重分布的 4-bit 量化表示。**

### `bf16`

**一种训练时更稳的 16-bit 计算精度。**

---

## 13. 一句话总结

如果只记住一个画面：

**LoRA / QLoRA 的核心不是“把整个大模型都训练一遍”，而是“冻结大模型，只训练一个小补丁；而 QLoRA 则进一步把这个冻结的大模型压缩存起来，从而让更大的模型也能在单卡上训练”。**

# 关键结论

这份文档只保留当前项目最值得记住的结论，不展开过程细节。

如果需要完整背景，见：

- [project-overview.md](/hy-tmp/llm-lab/docs/overview/project-overview.md)

## 1. 任务与协议

- 当前任务是一个**强结构化**的中文训练 / 部署问题诊断任务。
- 最终输出必须是严格 JSON，包含固定字段与枚举约束。
- 在这个任务里，`strict_json_prompt` 已经明确成为正式主线协议。

## 2. 5060Ti 阶段

- `0.5B` 的主要价值是流程验证，而不是最终效果。
- 这一阶段最重要的成果是：
  - 跑通数据构建
  - 跑通 SFT
  - 跑通自动评测
  - 跑通 LLM-as-a-judge
  - 跑通 DPO 候选数据采集

## 3. 模型规模：0.5B -> 3B -> 4B

- `3B` 明显优于 `0.5B`
- `4B strict lora` 明显优于 `3B strict lora`
- 因此在当前任务上，至少到 `4B` 这一档，继续 scale up 是值得的

## 4. 4090 阶段：4B baseline

- `4B strict` 已在 4090 上稳定复现
- 这说明当前项目已经进入正式主实验阶段，而不是单纯流程验证阶段

## 5. 4090 阶段：8B

- `8B QLoRA` 在训练信号上优于 `4B`
- 在正确 no-think 条件下，`8B base` 的 strict JSON 遵循能力很强
- `8B qlora` 相比 `4B strict lora` 的内容质量仍然更强
- `8B` 是当前训练、评测、judge、成本之间最平衡的主实验位

## 6. 4090 阶段：14B

- `14B QLoRA` 在单卡 4090 上训练可行
- 完整 `SFT` 的训练信号明显优于 `8B`
- 但在推理与评测阶段，开始明显碰到当前软件栈与基础设施边界
- 因此 `14B` 更像：
  - 当前资源约束下的训练甜点位上沿
  - 而不是稳定可用的主评测 / 主推理位

## 7. 任务对齐 vs 模型规模

- 在当前任务上，**任务对齐的收益足以跨越一定的模型规模差距**
- `4B strict SFT / DPO` 在 pairwise winner 上都能明显压过 `8B base`
- 这说明：
  - 已对齐的小模型
  - 可以在具体任务上稳定打过未对齐的大 base 模型

## 8. vLLM 与推理基础设施

- `vLLM` 在更小模型上已经显示出明显速度优势
- 但不同 backend 的输出行为并不天然一致
- 到 `14B` 这一档时，问题不再只是“快不快”，而是：
  - HF 路线开始脆弱
  - vLLM 版本 / CUDA / Qwen3 config 兼容性开始变成现实问题

## 9. DPO 数据

- 现有 DPO 数据主要来自：
  - train prompt
  - 双模型候选答案
  - judge 选择 winner
- 最终形成了两版可训练数据：
  - `high_only`
  - `high_plus_medium`
- 这批数据已经足够支撑第一版 DPO 探索

## 10. DPO 训练

- `4B` 和 `8B` 的 DPO 都能学到有效偏好信号
- `4B` 对当前 DPO 数据的训练响应更强
- `8B` 的 DPO 更像在强基线上的温和精修

## 11. DPO 的收益形态

- DPO 不是完全无效
- 但当前收益有限，且比 SFT 更依赖数据质量与偏好目标清晰度
- 自动规则评测并不能完整反映 DPO 收益
- pairwise judge 更能体现 DPO 在工程化质量上的改进

## 12. 当前最稳的主线判断

- 如果以“完整闭环、稳定评测、成本可控”为优先：
  - 当前主线应以 `8B` 为核心
- 如果以“训练上限探索”为优先：
  - `14B` 是当前甜点位上沿候选
- 如果以“继续提高工程化质量”为优先：
  - 后续主要瓶颈已经从模型训练转向偏好数据工程与推理基础设施

## 13. 为什么现在可以收束

- 项目已经完成：
  - 数据构建
  - SFT
  - 自动评测
  - judge
  - DPO 数据生成
  - DPO 训练
  - 更大模型探索
  - 推理基础设施边界定位

- 继续往下的主要成本将更多转向：
  - 高质量偏好数据定义
  - 更复杂的推理系统适配

对于一个以学习为主的项目，当前已经是一个合理且有价值的停点。

# 项目总览

这个项目围绕一个具体任务展开：

**构建一个中文“训练 / 部署问题诊断助手”，并用数据构建、SFT、评测、judge、DPO 与推理基础设施实验，逐步判断模型规模、对齐方式和工程边界。**

项目目标从一开始就不是“纯刷榜”或“单次跑通”，而是：

1. 跑通一条完整实验闭环
2. 找到适合当前任务的正式协议主线
3. 判断模型继续 scale up 是否仍有收益
4. 观察 DPO 在当前任务上是否值得继续投入
5. 明确当前基础设施的现实边界

## 1. 任务定义

当前任务输入通常包含：

- 用户问题
- 环境信息
- 命令
- 报错日志

目标输出是一个严格结构化的 JSON，对应一个工程诊断结果，核心字段包括：

- `category`
- `severity`
- `summary`
- `root_cause`
- `missing_info`
- `next_steps`

这意味着项目从一开始就不是开放式聊天，而是：

- 结构约束强
- 工程语义重
- 既要求格式稳定，也要求内容判断可靠

## 2. 整体实验路径

整个项目可以分成三个连续阶段：

### 阶段一：5060Ti / 流程验证与主线收束

这一阶段完成了：

- 数据构建
- LLaMA-Factory 数据视图转换
- `0.5B / 3B / 4B` 的 SFT 训练
- 自动规则评测
- pairwise judge
- DPO 候选采集链路

这个阶段最重要的结论是：

- `0.5B` 适合作为流程验证模型
- `strict_json_prompt` 明确优于更松散的默认 prompt 形态，正式收束成主线协议
- `3B` 比 `0.5B` 明显更强
- `4B strict` 比 `3B strict` 更强

换句话说，这一阶段把项目从：

- “这条链路能不能跑”

推进到了：

- “已经有一条清晰可持续的正式实验主线”

## 3. 阶段二：4090 / 更大模型与基础设施边界

在 `4090` 阶段，项目重点转向：

- 更大模型的 `QLoRA`
- 在正式协议下的真实边际收益
- 单卡推理与评测基础设施的现实承载能力

这一阶段主要做了：

### 4B

- 在 4090 上稳定复现 `4B strict` baseline
- 确认训练、HF 推理评测链路可用

### 8B

- 用 `8B QLoRA` 做正式主实验
- 在训练信号上继续优于 4B
- 在正确 no-think 配置下，base 的 strict JSON 遵循能力很强
- `8B qlora` 在内容质量上相对 `4B strict lora` 保持稳健领先
- 同时完成了一轮吞吐优化：
  - `batch_size 1 -> 2`
  - `grad_acc 32 -> 16`
  - 在几乎不损失效果的前提下显著提升训练效率

### 14B

- 在单卡 `4090 + QLoRA` 条件下稳定训练成功
- 训练信号明显强于 `8B`
- 但 HF / PEFT / vLLM 推理基础设施开始显著暴露边界：
  - `HF + PEFT + auto device_map` 装载问题
  - 强制单卡后又出现显存边界
  - 老版本 `vLLM` 与新版 `Qwen3-14B` 配置结构不兼容

因此 `14B` 当前阶段的最准确定位是：

- **训练甜点位上沿候选**
- 但还不是当前这套基础设施上“可无缝接管主线的稳定推理位”

## 4. 阶段三：DPO / 偏好优化探索

在 `DPO` 阶段，项目没有直接套用现成通用偏好数据，而是：

- 从现有 `train` 分布样本出发
- 用两个候选模型对同一 prompt 生成答案
- 用一个独立 judge 模型做 pairwise 判优
- 再清洗成适合 LLaMA-Factory 的 preference 数据集

最终得到的第一版数据集有两层：

- `high_only`
- `high_plus_medium`

这个阶段的重要发现不是“DPO 显著提升一切”，而是：

### 第一层发现

`DPO` 是可行的。

- `4B` 和 `8B` 都能从当前高置信偏好数据中学到有效信号
- 在 pairwise judge 中都表现出相对 SFT 的正向收益

### 第二层发现

`DPO` 的收益是有限的，而且形态会随模型规模变化。

- `4B DPO` 更像被明显“拉升”
- `8B DPO` 更像在已有较强基线上做“精修”

### 第三层发现

当前任务里的偏好目标是复合的：

- groundedness
- root cause quality
- actionability
- missing_info quality
- overall engineering quality

这比很多典型 RLHF / DPO 任务（如安全性、礼貌性、风格）更难做成单轴、稳定的大收益偏好优化。

所以当前最稳的判断是：

- DPO 在这个任务上**有用**
- 但主要瓶颈已经从“训练能不能跑”转向：
  - 偏好目标是否足够清晰
  - pair 是否足够纯
  - 数据工程是否值得继续投入

## 5. 当前阶段的最重要结论

如果只保留几条结论，当前项目已经比较清楚地说明：

1. `strict_json_prompt` 是当前任务的正式协议主线
2. 训练后的中小模型在这个任务上，能明显优于未对齐的大 base 模型
3. `8B` 是当前训练、评测、judge、工程成本最平衡的主实验位
4. `14B` 训练收益仍然存在，但基础设施边界已经明显出现
5. DPO 可行，但当前收益有限、解释难度更高，后续若继续投入，重点将是偏好数据工程而不是单纯增加训练轮数

## 6. 为什么当前可以阶段性收束

从“以学习为主的项目”角度看，当前已经完成了：

- 数据构建
- SFT
- 自动评测
- judge
- DPO 数据生成与训练
- 更大模型 scale-up
- 推理基础设施尝试与边界定位

这意味着项目已经不仅是：

- 跑通了一个模型训练脚本

而是跑通并分析了一整条实验闭环。

再往下继续投入，主要成本会更多转向：

- 高质量偏好数据设计
- 更复杂的推理基础设施适配
- 更细的超参和实验管理

这些当然也有价值，但对当前目标来说，边际收益已经明显下降。

## 7. 一句话总结

这个项目最终不是在回答“哪个模型分数最高”，而是在回答：

**对于一个强结构化、工程语义重的诊断任务，数据、协议、模型规模、对齐方式和推理基础设施之间，到底怎样互相制约。**

而到目前为止，项目已经给出了足够清楚、也足够有价值的阶段性答案。
